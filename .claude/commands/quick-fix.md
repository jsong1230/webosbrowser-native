## 사용 기준
- 버그 수정, 텍스트/설정 변경, 스타일 수정, 린트 에러 수정 등
- 새로운 기능 추가가 아닌, 기존 기능의 수정/보완
- 분석서·설계서·계획서가 필요 없는 수준의 작업

## Step 1: 현황 파악
- 관련 코드/파일 확인
- 이슈 원인 분석 (버그인 경우)

## Step 2: 수정
- 해당 에이전트(backend-dev 또는 frontend-dev)로 코드 수정
- 영향 범위가 backend + frontend 양쪽이면 순차 수정

## Step 3: 테스트 (test-runner 에이전트)
- 기존 테스트 통과 확인
- 수정된 부분 관련 테스트 추가 (필요 시)

## Step 4: 리뷰 (선택)
- 변경 범위가 크면 code-reviewer 에이전트로 리뷰
- 단순 변경이면 생략 가능

## Step 5: Git 커밋
- `fix: {수정 내용}` 형식으로 커밋

수정 내용: $ARGUMENTS
