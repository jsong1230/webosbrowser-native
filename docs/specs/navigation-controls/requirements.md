# 페이지 탐색 컨트롤 — 요구사항 분석서

## 1. 개요

### 기능명
F-04: 페이지 탐색 컨트롤 (NavigationBar)

### 목적
웹 브라우저의 기본 탐색 기능(뒤로 가기, 앞으로 가기, 새로고침, 홈으로 이동)을 리모컨 최적화 UI로 제공하여, 사용자가 프로젝터 대화면 환경에서 직관적으로 웹 페이지 간 이동을 할 수 있도록 합니다.

### 대상 사용자
- webOS Browser의 모든 사용자
- 리모컨으로 웹 페이지를 탐색하는 일반 사용자
- 여러 웹 페이지를 오가며 정보를 찾는 사용자
- 영상 스트리밍 플랫폼 내에서 콘텐츠 간 이동하는 사용자

### 요청 배경
- 웹 브라우저의 핵심 네비게이션 기능(뒤로/앞으로)은 브라우저 사용성의 기본 요소입니다.
- F-02(웹뷰 통합)에서 웹 페이지 렌더링 기능을 구현했으므로, 이를 제어할 탐색 UI가 필요합니다.
- 리모컨 환경에서는 브라우저 메뉴가 없으므로 명시적인 버튼 UI가 필수입니다.
- PRD에서 "리모컨 최적화 UX"가 핵심 가치로 정의되었으며, 탐색 버튼은 리모컨으로 쉽게 접근 가능해야 합니다.

## 2. 기능 요구사항 (Functional Requirements)

### FR-1: 뒤로 가기 버튼
- **설명**: 현재 웹뷰의 히스토리 스택에서 이전 페이지로 이동하는 버튼
- **유저 스토리**:
  - As a 웹 브라우저 사용자, I want 뒤로 가기 버튼을 클릭하여 이전에 본 페이지로 돌아가고 싶다, so that 잘못 클릭한 링크나 이전 내용을 다시 확인할 수 있다.
- **상세 요구사항**:
  - 버튼 클릭 시 iframe의 history.back() 호출
  - 히스토리 스택이 비어있을 때(첫 페이지) 버튼 비활성화 (disabled)
  - 비활성화된 버튼은 시각적으로 구분 (흐릿하게 표시, opacity 적용)
  - 리모컨 포커스를 받을 수 있으나 클릭 시 동작하지 않음
  - 아이콘: 왼쪽 화살표 (← 또는 ⬅️)
- **우선순위**: Must

### FR-2: 앞으로 가기 버튼
- **설명**: 뒤로 가기 이후 다시 앞 페이지로 이동하는 버튼
- **유저 스토리**:
  - As a 웹 브라우저 사용자, I want 뒤로 간 후 다시 앞으로 가기 버튼을 클릭하여 최근 페이지로 돌아가고 싶다, so that 탐색 중 실수로 뒤로 간 경우 빠르게 복구할 수 있다.
- **상세 요구사항**:
  - 버튼 클릭 시 iframe의 history.forward() 호출
  - 앞으로 갈 히스토리가 없을 때 버튼 비활성화
  - 비활성화된 버튼은 시각적으로 구분 (FR-1과 동일한 스타일)
  - 아이콘: 오른쪽 화살표 (→ 또는 ➡️)
- **우선순위**: Must

### FR-3: 새로고침 버튼
- **설명**: 현재 페이지를 다시 로드하는 버튼
- **유저 스토리**:
  - As a 웹 브라우저 사용자, I want 새로고침 버튼을 클릭하여 페이지를 다시 불러오고 싶다, so that 콘텐츠가 업데이트되거나 로딩 실패 시 재시도할 수 있다.
- **상세 요구사항**:
  - 버튼 클릭 시 iframe의 location.reload() 호출
  - 버튼은 항상 활성화 상태 (비활성화되지 않음)
  - 로딩 중에도 클릭 가능 (로딩 중단 후 재로드)
  - 아이콘: 새로고침 아이콘 (🔄 또는 원형 화살표)
- **우선순위**: Must

### FR-4: 홈 버튼
- **설명**: 사용자가 설정한 홈페이지 URL로 이동하는 버튼
- **유저 스토리**:
  - As a 웹 브라우저 사용자, I want 홈 버튼을 클릭하여 자주 사용하는 시작 페이지로 바로 이동하고 싶다, so that 깊은 탐색 후 빠르게 시작점으로 돌아갈 수 있다.
- **상세 요구사항**:
  - 버튼 클릭 시 설정된 홈페이지 URL로 WebView 로드
  - 기본 홈페이지 URL: `https://www.google.com` (설정 없을 경우)
  - 홈페이지 설정은 F-11(설정 화면)에서 변경 가능 (현재는 하드코딩)
  - 버튼은 항상 활성화 상태
  - 아이콘: 집 아이콘 (🏠 또는 ⌂)
- **우선순위**: Must

### FR-5: 리모컨 포커스 네비게이션
- **설명**: 리모컨 방향키로 버튼 간 포커스 이동이 자연스럽게 동작
- **유저 스토리**:
  - As a 웹 브라우저 사용자, I want 리모컨 좌/우 방향키로 탐색 버튼 간 이동하고 싶다, so that 마우스 없이도 원하는 버튼을 선택할 수 있다.
- **상세 요구사항**:
  - NavigationBar의 버튼들은 Enact Spotlight로 관리되어 포커스 가능
  - 버튼 순서: 뒤로 → 앞으로 → 새로고침 → 홈 (좌→우)
  - 리모컨 좌/우 방향키로 버튼 간 포커스 이동
  - 리모컨 위 방향키로 WebView로 포커스 이동
  - 포커스된 버튼은 시각적으로 강조 (테두리, 배경색 변경)
  - 리모컨 선택 버튼(Enter/OK)으로 버튼 활성화
- **우선순위**: Must

### FR-6: 버튼 상태 동기화
- **설명**: WebView의 히스토리 상태에 따라 버튼 활성/비활성 상태가 자동으로 업데이트
- **유저 스토리**:
  - As a 웹 브라우저 사용자, I want 뒤로 갈 수 없을 때 뒤로 버튼이 비활성화되는 것을 보고 싶다, so that 클릭해도 동작하지 않을 버튼을 미리 알 수 있다.
- **상세 요구사항**:
  - WebView가 onNavigationChange 이벤트를 발생시킬 때 히스토리 상태 확인
  - history.length, 현재 인덱스를 기반으로 뒤로/앞으로 버튼 활성 여부 결정
  - 버튼 활성 여부는 React 상태로 관리
  - 상태 변경 시 UI 즉시 업데이트 (지연 없음)
- **우선순위**: Must
- **기술적 제약**: iframe의 Same-Origin Policy로 인해 다른 도메인 페이지에서는 history 접근 제한. 이 경우 버튼은 항상 활성화 상태로 표시하고, 클릭 시 동작 실패 시 무시 처리.

### FR-7: 리모컨 Back 키 매핑
- **설명**: 리모컨의 물리적 Back 버튼을 누를 때 브라우저 뒤로 가기 동작
- **유저 스토리**:
  - As a 웹 브라우저 사용자, I want 리모컨의 뒤로 버튼을 눌러 페이지를 뒤로 가고 싶다, so that UI 버튼을 찾지 않고도 빠르게 뒤로 갈 수 있다.
- **상세 요구사항**:
  - BrowserView 레벨에서 onKeyDown 이벤트 캡처
  - KeyCode 확인 (Backspace 또는 webOS Back 키)
  - WebView에 포커스가 있을 때: 브라우저 히스토리 뒤로 (FR-1과 동일)
  - WebView에 포커스가 없을 때(URLBar, NavigationBar): Spotlight 포커스 이탈 (기본 동작)
  - 히스토리가 비어있을 때: 앱 종료 또는 홈 화면 이동 (F-15 구현 시 연동)
- **우선순위**: Must

## 3. 비기능 요구사항 (Non-Functional Requirements)

### NFR-1: 성능
- **버튼 응답 시간**: 버튼 클릭 후 0.3초 이내에 페이지 네비게이션 시작 (PRD 기준)
- **히스토리 상태 업데이트**: 페이지 로드 완료 후 0.5초 이내에 버튼 상태 업데이트
- **메모리 사용**: NavigationBar 컴포넌트는 10KB 이하의 메모리만 사용 (경량 UI)

### NFR-2: UX (리모컨 최적화)
- **버튼 크기**: 최소 100px x 80px (3m 거리에서 인식 가능, PRD 기준)
- **버튼 간격**: 최소 20px (리모컨 포커스 이동 시 명확한 시각적 구분)
- **포커스 표시**: 포커스된 버튼은 2px 두께의 흰색 테두리 또는 배경색 변경
- **아이콘 크기**: 최소 32px x 32px (명확한 인식)
- **폰트 크기**: 버튼 라벨은 최소 16px (PRD 기준, 선택 사항)
- **색상 대비**: WCAG 2.1 AA 기준 (4.5:1 이상) 준수

### NFR-3: 접근성
- **비활성화된 버튼 표시**: 비활성 버튼은 opacity: 0.5 또는 회색으로 표시
- **시각적 피드백**: 버튼 클릭 시 눌림 효과 (scale 애니메이션 또는 배경색 변경)
- **키보드 네비게이션**: 리모컨 방향키로 모든 버튼 접근 가능
- **스크린 리더 지원**: aria-label, aria-disabled 속성 추가 (선택 사항, webOS 지원 제한)

### NFR-4: 신뢰성
- **에러 처리**: iframe history API 호출 실패 시 조용히 실패 (에러 메시지 없음, 콘솔 로그만)
- **상태 복구**: 히스토리 상태 확인 실패 시 버튼은 기본 활성화 상태로 설정
- **CORS 대응**: Same-Origin Policy로 히스토리 접근 실패 시 대체 로직 사용 (버튼 항상 활성화)

### NFR-5: 호환성
- **webOS 버전**: webOS 4.x 이상 (LG 프로젝터 HU175QW)
- **Enact 버전**: @enact/moonstone 4.x 이상
- **브라우저 호환**: iframe의 history API는 모든 webOS 버전에서 지원

## 4. 제약사항

### 기술적 제약
1. **iframe의 Same-Origin Policy**:
   - 다른 도메인 페이지의 history 객체에 직접 접근 불가
   - history.length, history.state 읽기 제한
   - 대응: 버튼 활성 여부를 WebView에서 전달받는 Props 기반 관리

2. **webOS 리모컨 키 이벤트**:
   - webOS는 리모컨 Back 버튼을 Backspace 키로 변환
   - 일부 webOS 버전은 커스텀 키 코드 사용
   - 대응: 키 코드 매핑 테이블 작성 및 테스트

3. **Enact Spotlight 포커스 관리**:
   - NavigationBar와 WebView 간 포커스 전환이 명확해야 함
   - 대응: BrowserView에서 포커스 흐름 설계 (F-02 설계서 참조)

### 의존성
- **F-01 (프로젝트 초기 설정)**: Enact 프로젝트 구조, Spotlight 설정 필요 (완료)
- **F-02 (웹뷰 통합)**: WebView 컴포넌트가 구현되어 있어야 함 (완료)
- **F-02의 WebView Props**: onNavigationChange 콜백이 구현되어 있어야 히스토리 상태 동기화 가능

### 플랫폼 제약
- **리모컨 입력**: 마우스/터치 입력 불가, 리모컨 전용
- **화면 크기**: 프로젝터 대화면 (1920x1080 이상) 기준 UI 설계
- **성능**: webOS 하드웨어 제약으로 복잡한 애니메이션 자제

## 5. 용어 정의

| 용어 | 정의 |
|------|------|
| **NavigationBar** | 뒤로/앞으로/새로고침/홈 버튼을 포함한 탐색 컨트롤 UI 컴포넌트 |
| **히스토리 스택** | 웹 브라우저가 관리하는 방문 페이지 기록 (history API) |
| **뒤로 가기** | history.back() 호출, 이전 페이지로 이동 |
| **앞으로 가기** | history.forward() 호출, 뒤로 간 후 다시 앞 페이지로 이동 |
| **새로고침** | location.reload() 호출, 현재 페이지 재로드 |
| **홈페이지** | 사용자가 설정한 기본 시작 페이지 (기본값: Google) |
| **리모컨 Back 키** | webOS 리모컨의 물리적 뒤로 버튼 (Backspace 키로 매핑) |
| **Spotlightable** | Enact Spotlight 시스템에서 리모컨 포커스를 받을 수 있는 컴포넌트 |
| **비활성화 (disabled)** | 버튼이 클릭 가능하지만 동작하지 않는 상태 (시각적 피드백만) |

## 6. 범위 외 (Out of Scope)

### 이번 기능에서 하지 않는 것
1. **URL 표시 기능**: NavigationBar는 현재 URL을 표시하지 않음 (F-03 URLBar에서 구현)
2. **로딩 진행률 표시**: 새로고침 시 로딩 상태는 F-05(로딩 인디케이터)에서 처리
3. **히스토리 패널 UI**: 방문 기록 목록은 F-08(히스토리 관리)에서 구현
4. **북마크 추가 버튼**: 북마크 기능은 F-07(북마크 관리)에서 구현
5. **설정 버튼**: 설정 화면 진입은 F-10(설정 화면)에서 구현
6. **탭 전환 버튼**: 탭 관리는 F-06(탭 관리)에서 구현
7. **진행률 기반 새로고침 중단**: iframe은 진행률 API를 제공하지 않으므로 미구현
8. **홈페이지 설정 UI**: F-11(설정 화면)에서 구현 예정, 현재는 하드코딩

### 향후 확장 가능 기능
1. **버튼 커스터마이징**: 사용자가 버튼 순서, 표시 여부를 설정 (F-11 연동)
2. **제스처 지원**: 리모컨 스와이프로 뒤로/앞으로 (webOS 6.x 이상)
3. **음성 명령**: "뒤로 가기"라고 말하면 동작 (추후 AI 통합)

## 7. 완료 기준 (Acceptance Criteria)

### AC-1: 버튼 기본 동작
- [ ] 뒤로 가기 버튼 클릭 시 이전 페이지로 이동
- [ ] 앞으로 가기 버튼 클릭 시 다음 페이지로 이동
- [ ] 새로고침 버튼 클릭 시 현재 페이지 재로드
- [ ] 홈 버튼 클릭 시 Google 홈페이지로 이동

### AC-2: 버튼 상태 관리
- [ ] 첫 페이지에서 뒤로 가기 버튼 비활성화
- [ ] 뒤로 간 후에만 앞으로 가기 버튼 활성화
- [ ] 비활성화된 버튼은 흐릿하게 표시 (opacity: 0.5)
- [ ] 비활성화된 버튼 클릭 시 동작하지 않음

### AC-3: 리모컨 네비게이션
- [ ] 리모컨 좌/우 방향키로 버튼 간 포커스 이동
- [ ] 포커스된 버튼에 흰색 테두리 표시
- [ ] 리모컨 선택 버튼으로 버튼 활성화
- [ ] 리모컨 위 방향키로 WebView로 포커스 이동

### AC-4: 리모컨 Back 키 매핑

## 8. 테스트 시나리오

### 시나리오 1: 정상 네비게이션 플로우
1. Google 홈페이지 로드 → 뒤로 버튼 비활성화 확인
2. "YouTube" 검색 → 검색 결과 페이지 로드 → 뒤로 버튼 활성화 확인
3. 뒤로 버튼 클릭 → Google 홈페이지로 복귀 → 앞으로 버튼 활성화 확인
4. 앞으로 버튼 클릭 → 검색 결과 페이지로 복귀
5. 홈 버튼 클릭 → Google 홈페이지로 복귀 → 앞으로 버튼 비활성화 확인

### 시나리오 2: 리모컨 Back 키 사용
1. Google 홈페이지 로드
2. "Naver" 검색 → 검색 결과 페이지 로드
3. WebView에 포커스 → 리모컨 Back 키 → Google 홈페이지로 복귀
4. 리모컨 아래 방향키 → NavigationBar로 포커스 이동
5. 리모컨 Back 키 → 포커스 이탈 (페이지 이동 안 함)

### 시나리오 3: 새로고침 동작
1. YouTube 홈페이지 로드
2. 비디오 하나 클릭 → 비디오 페이지 로드
3. 새로고침 버튼 클릭 → 비디오 페이지 재로드 → 동일 비디오 표시 확인

### 시나리오 4: 에러 대응
1. 네트워크 끊기 → "example.com" 로드 실패
2. 뒤로 버튼 클릭 → 정상 동작 (이전 페이지로 복귀)
3. 네트워크 복구 → 새로고침 버튼 클릭 → 페이지 정상 로드

### 시나리오 5: CORS 제약 상황
1. Same-Origin이 아닌 외부 사이트(예: YouTube) 로드
2. 버튼 상태 확인 → 히스토리 접근 실패 시에도 버튼 활성화 상태 유지
3. 버튼 클릭 → 정상 동작 확인 (히스토리 상태는 iframe 내부에서 관리)
